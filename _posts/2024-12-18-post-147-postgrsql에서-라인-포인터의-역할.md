관계형 데이터베이스(RDB)에서 데이터는 물리 디스크와 메모리 사이에서 블록 혹은 **페이지(Page)** 단위로 관리됩니다. PostgreSQL의 경우 기본 페이지 크기는 **8KB**이며, 이 제한된 크기 내에서 극도로 조밀한 저장 공간 설계를 실현하고 있습니다. 

이번 포스팅에서는 PostgreSQL 공유 버퍼와 물리 파일의 기본 단위인 페이지의 내부 레이아웃을 살펴보고, 그중에서도 데이터 접근의 나침반 역할을 하는 **라인 포인터(Line Pointer / Item Pointer)**의 핵심 개념에 대해 집중 분석해 보겠습니다.

---

## 1. PostgreSQL 페이지(Page) 레이아웃 구조

PostgreSQL의 8KB 페이지는 크게 네 가지 영역으로 양분되어 위아래로 자라나는 지능적인 구조를 취하고 있습니다.

![](/images/147/img.png)

1. **Page Header (페이지 헤더)**: 
   * 페이지의 메타데이터(LSO, 빈 공간 시작점, 사용 가능한 공간 끝점 등)를 담고 있으며 크기는 24바이트입니다.
2. **Line Pointers (라인 포인터 영역)**:
   * 페이지 헤더 바로 뒤에 위치하며, 실제 튜플(Tuple / 로우 데이터)이 저장된 정확한 메모리 오프셋 주소와 데이터 길이를 가리킵니다.
   * 라인 포인터는 개당 **4바이트** 크기를 차지하며, 데이터가 입력될 때마다 **위에서 아래로(Top-to-Bottom)** 늘어납니다.
3. **Free Space (여유 공간)**:
   * 라인 포인터 영역과 튜플 데이터 영역 사이에 존재하는 비어있는 임시 공간입니다.
4. **Tuple Data (실제 데이터 영역)**:
   * 실제 행(Row) 정보들이 영구 기록되는 영역으로, 페이지의 맨 하단 끝부분부터 **아래에서 위로(Bottom-to-Top)** 채워집니다.

---

## 2. 라인 포인터(Line Pointer)의 결정적인 역할

라인 포인터는 실제 데이터 튜플의 간접 도우미(Indirection Pointer) 역할을 함으로써 데이터 조회 및 갱신 성능을 극적으로 끌어올립니다.

### 1) 데이터 탐색 경로의 간소화 (TID)
PostgreSQL에서 모든 인덱스(Index)는 실제 데이터 튜플의 디스크 주소를 직접 참조하지 않습니다. 인덱스는 대신 튜플의 식별자인 **TID(Tuple Identifier)**를 저장합니다.
* TID는 **`(블록 번호, 라인 포인터 인덱스)`** 쌍으로 구성됩니다. (예: `(0, 1)` -> 0번째 블록의 1번째 라인 포인터)
* 인덱스를 스캔할 때 실제 튜플의 물리적 위치(Offset)가 바뀌더라도, 라인 포인터가 가리키는 주소만 페이지 헤더 내에서 업데이트해주면 되기 때문에 인덱스까지 도미노처럼 갱신할 필요가 없어 쓰기 성능을 보장할 수 있습니다.

### 2) 데이터가 삭제되거나 수정(MVCC)될 때의 거동
PostgreSQL은 다중 버전 동시성 제어(MVCC) 메커니즘을 따르므로, `UPDATE` 시 기존 튜플을 지우지 않고 신규 튜플을 페이지의 빈 공간에 추가로 적재합니다.
* **삭제 시**: 해당 튜플을 가리키던 라인 포인터는 'Dead' 상태로 마킹됩니다.
* **VACUUM 구동 시**: 가비지 컬렉터인 VACUUM이 작동하여 Dead 튜플의 물리적 데이터 공간을 회수합니다. 이때 라인 포인터는 비어있다는 식별 플래그를 가지게 되며, 추후 새로운 행이 들어올 때 재사용(Reuse)됩니다.

### 3) HOT (Heap-Only Tuple) 최적화의 디딤돌
동일 페이지 내에서 `UPDATE`가 일어날 때, 인덱스 엔트리를 새로 파지 않고 기존 라인 포인터에서 신규 튜플을 가리키는 라인 포인터로 페이지 내부 포인터 연결(Chaining)을 맺어주는 **HOT 최적화**도 이 간접 포인팅 구조 덕분에 가능합니다.

라인 포인터는 단순한 오프셋 주소록이 아닌, PostgreSQL의 MVCC 구현체와 디스크 단편화 해소를 정교하게 떠받치고 있는 아키텍처의 핵심 빌딩 블록입니다.
