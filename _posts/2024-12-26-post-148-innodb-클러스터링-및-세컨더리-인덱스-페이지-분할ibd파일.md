MySQL의 기본 스토리지 엔진인 **InnoDB**는 디스크 상의 테이블 데이터를 정교한 B-Tree 인덱스 구조로 관리합니다. 특히 물리적 저장 구조 파일인 `.ibd` 파일 내부에서는 테이블의 모든 행이 인덱스 노드의 한 부분으로 동작합니다.

이번 포스팅에서는 InnoDB의 근간이 되는 **클러스터형 인덱스(Clustered Index)**와 **보조 인덱스(Secondary Index)**의 상호 주소 참조 구조를 분석하고, 데이터가 다량 적재될 때 발생하는 성능 저하의 주범인 **페이지 분할(Page Split)** 메커니즘에 대해 자세히 알아보겠습니다.

---

## 1. InnoDB의 인덱스 물리 구조

InnoDB 스토리지 엔진에서 테이블은 최소 16KB 크기의 **페이지(Page)**들의 집합으로 관리됩니다.

### 1) 클러스터형 인덱스 (Clustered Index)
* **특징**: 테이블당 단 하나만 생성될 수 있으며, 보통 기본키(PK)를 기준으로 결정됩니다.
* **물리 구조**: B-Tree의 **단말 노드(Leaf Node)에 실제 행 데이터(레코드)가 PK 순서대로 정렬되어 직접 저장**됩니다. 즉, 인덱스 그 자체가 곧 테이블의 물리적인 데이터 정렬 구조와 동일합니다.

### 2) 보조 인덱스 (Secondary Index)
* **특징**: 기본키 외에 추가로 생성한 일반 인덱스들입니다.
* **물리 구조**: 보조 인덱스의 단말 노드(Leaf Node)에는 실제 행의 물리적인 디스크 주소(Pointer)가 기록되지 않습니다. 대신, 해당 행의 **기본키(PK) 값**이 담겨 있습니다.
* **이유**: 만약 보조 인덱스가 튜플의 실제 물리 주소를 직접 포인팅하고 있다면, 페이지 분할 등으로 인해 데이터의 물리 위치가 바뀔 때마다 테이블에 정의된 모든 보조 인덱스의 리프 노드 주소를 전부 뒤쫓아가 갱신해야 하는 어마어마한 쓰기 지연(Write Amplification)이 발생하기 때문입니다.

## 2. 데이터 유입에 따른 인덱스 페이지의 변화 과정

데이터가 순차적 혹은 무작위로 입력됨에 따라 `.ibd` 파일 내부의 인덱스 페이지들은 아래 그림과 같이 역동적으로 생성되고 나뉩니다.

### 1) 테이블 및 인덱스 최초 생성 단계
초기 테이블이 생성되면 인덱스를 관리할 최초의 루트 페이지가 확보됩니다.
* **클러스터형 인덱스**: 최초 데이터들이 단일 리프 페이지 내에 조밀하게 적재됩니다.
* **보조 인덱스**: 인덱스를 타겟팅하는 별도의 세컨더리 인덱스 리프 페이지가 나란히 열립니다.

### 2) 대량의 데이터 삽입과 리프 노드의 분화
많은 양의 행이 추가되면서 하나의 16KB 페이지 용량이 한계에 다다르게 됩니다.
* **클러스터형 인덱스**: 기존 리프 노드에 공간이 부족해지면 리프 노드가 2개로 분할되고, 기존 루트 노드는 분열된 자식 리프 페이지들을 참조하는 중간 브랜치(Non-leaf) 주소 노드로 승격됩니다.

## 3. 페이지 분할(Page Split)의 부하 메커니즘

만약 기본키(PK)가 순차적으로 증가하는 auto_increment 형식이 아니라 UUID 등 무작위 난수 기반이거나, 중간 인덱스 위치에 데이터를 무작위로 끼워 넣어야(Insert) 하는 상황을 가정해 보겠습니다.

### 페이지 분할의 발생 프로세스
1. **페이지 포화**: 임의의 리프 페이지 중간에 데이터를 삽입해야 하는데, 해당 16KB 페이지의 빈 공간(Free Space)이 전혀 없는 상태입니다.
2. **신규 페이지 할당**: InnoDB는 새로운 빈 페이지를 한 개 더 할당받습니다.
3. **데이터 복사 및 정렬**: 포화 상태였던 기존 페이지의 전체 레코드 중 절반 가량을 떼어내 신규 페이지로 마이그레이션하고, 신규 삽입 행을 위치에 맞게 삽입합니다.
4. **상위 노드 갱신**: 분할이 완료되면 상위 루트/브랜치 노드에 신규 페이지를 참조하는 키 링크 정보를 반영합니다.

### 페이지 분할이 성능에 미치는 영향
* **디스크 쓰기 오버헤드**: 페이지가 쪼개지면서 단지 1개의 행을 추가하기 위해 기존 데이터 페이지를 읽고 두 개의 다른 물리 페이지에 쓰는 대량의 I/O 작업이 수반됩니다.
* **페이지 내부 단편화**: 페이지 분할이 잦을수록 각 페이지 내부에 데이터를 꽉 채우지 못하고 빈 공간이 많이 발생하여, 인덱스 전체의 깊이(Depth)가 깊어지고 디스크 낭비가 초래됩니다.

따라서 대량의 데이터 쓰기가 동반되는 고성능 시스템일수록 기본키를 순차 정렬 키로 설계하여 페이지 분할(Page Split) 오버헤드를 사전에 예방하는 것이 극도로 중요합니다.
